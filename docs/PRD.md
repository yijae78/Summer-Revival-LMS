# PRD (Product Requirements Document)
# FLOWING — 여름행사 Learning Management System - 제품 요구사항 명세서

> **프로젝트명**: FLOWING — 여름행사 Learning Management System
> **버전**: v1.0
> **작성일**: 2026-03-13
> **배포 방식**: 웹 (Vercel)

## 1. 프로젝트 개요

### 1.1 배경
한국 교회(장로교) 여름행사(수련회, 성경학교, 캠프 등) 운영 시, 참가자 등록부터 출석 관리, 일정 안내, 자료 배포까지 구글 폼 + 엑셀 + 카카오톡 단체방을 조합하여 분산 운영하고 있음. 이를 **올인원 대시보드 형태의 웹 LMS**로 통합하여 운영 효율을 극대화함.

### 1.2 목표
- 교회 여름행사 **등록 → 운영 → 정리**까지 전 과정을 하나의 플랫폼에서 관리
- **BYOS (Bring Your Own Supabase)**: 각 교회가 자체 Supabase를 연결하여 **데이터 소유권 확보**, 앱 제공자에게 비용 부담 없음
- **앱 내 온보딩 위자드**로 Supabase 연결을 단계별 안내 (IT 비전문가도 5분 내 설정)
- **모바일 우선** 설계로 현장에서 즉시 활용 가능
- **매년 재사용** 가능한 템플릿 기반 설계

### 1.3 성공 지표
| 지표 | 목표 |
|------|------|
| 참가자 등록률 (온라인) | 90% 이상 |
| 현장 출석 체크 소요 시간 | 세션당 3분 이내 |
| 모바일 접근 비율 | 80% 이상 |
| 관리자 만족도 | 4.0/5.0 이상 |

## 2. 타겟 사용자

### 2.1 사용자 역할 정의

| 역할 | 설명 | 예상 인원 | 주요 디바이스 |
|------|------|-----------|--------------|
| **관리자 (Admin)** | 교역자 (담당 목사/전도사) | 2~5명 | PC + 모바일 |
| **교사 (Staff)** | 조장, 부장교사, 봉사자 | 10~30명 | 모바일 (주) |
| **참가자 (Student)** | 수련회/성경학교 참가 학생 | 50~200명 | 모바일 전용 |
| **학부모 (Parent)** | 참가자 보호자 | 30~150명 | 모바일 전용 |

### 2.2 사용자 스토리

**관리자로서:**
- 행사 정보를 등록하고 전체 현황을 대시보드에서 한눈에 보고 싶다
- 참가자 등록 현황과 참가비 수납 상태를 실시간으로 확인하고 싶다
- 조/반 편성을 쉽게 하고, 교사를 배정하고 싶다
- 행사 후 출석/참여 통계를 뽑아 보고서를 만들고 싶다

**교사로서:**
- 담당 조의 학생 목록과 연락처를 모바일에서 바로 확인하고 싶다
- 매 세션마다 출석을 빠르고 간편하게 체크하고 싶다
- 오늘의 일정과 교재를 한 곳에서 확인하고 싶다
- 조별 활동 점수를 쉽게 입력하고 싶다

**참가자로서:**
- 오늘 일정과 내 조 정보를 한눈에 보고 싶다
- 퀴즈에 참여하고 내 점수/순위를 확인하고 싶다
- 사진/영상을 쉽게 볼 수 있으면 좋겠다
- 복잡한 가입 없이 바로 접속하고 싶다

**학부모로서:**
- 자녀의 참가 신청을 편하게 하고 싶다
- 행사 일정과 공지사항을 확인하고 싶다
- 행사 중 사진을 볼 수 있으면 좋겠다

## 5. 비기능적 요구사항

### 5.1 성능
- 페이지 로드: 3초 이내 (3G 환경 기준)
- 출석 체크 응답: 1초 이내
- 동시 접속: 200명 이상 지원

### 5.2 오프라인 지원 (PWA) - 수련원 네트워크 대응
- 일정표, 교재, 조원 목록은 오프라인 캐싱 (Serwist Network First + Cache Fallback)
- 출석 데이터는 IndexedDB 큐에 로컬 저장 → 온라인 복귀 시 자동 동기화
- 옵티미스틱 UI: 출석 체크 즉시 반영, 서버 확인 후 실패 시 롤백
- 오프라인 배너: 상단 amber 알림 (slide-down), 온라인 복귀 시 green 토스트
- PWA 설치 프롬프트: 첫 성공 액션 후 제안 (Android), iOS는 수동 설치 가이드
- 사전 캐싱: 양호한 WiFi에서 첫 접속 시 핵심 데이터 일괄 프리페치
- 상세: `docs/UX-PATTERNS.md` 섹션 6, 8 참조

### 5.3 보안/개인정보 (개인정보보호법 PIPA 준수)
- HTTPS 필수
- 개인정보 수집·이용 동의 (제15조)
- 만 14세 미만 법정대리인 동의 (제22조 제6항)
- 민감정보(건강/알레르기) 별도 동의 (제23조)
- 사진/초상권 게시 동의 (별도 수집)
- 개인정보 국외 이전(Supabase/미국) 동의 (제28조의8)
- 행사 종료 후 단계적 데이터 파기 정책 (제21조)
- 상세: `docs/SECURITY.md` 참조

### 5.4 접근성 (다연령 포용)
- **WCAG 2.2 AA** (ISO/IEC 40500:2025) 기준 준수
- 최소 텍스트: 16px (본문), 13px (캡션), 12px (오버라인). **10px 이하 완전 제거**
- 터치 타깃: **48x48px** 이상 (WCAG 2.5.8), 최소 간격 8px
- 한글 행간: `line-height: 1.7`, `word-break: keep-all`
- 색상 대비: 4.5:1 이상, Slate 400은 장식용만 (본문 Slate 500+)
- Reduced Motion: `prefers-reduced-motion` 지원, 인앱 토글 제공
- 다크 모드: 시스템 자동 감지 + 수동 토글
- 빈 상태: 모든 Empty State에 아이콘 + 설명 + CTA 포함
- 상세: `docs/ACCESSIBILITY.md` 참조

### 5.5 한국형 UX
- 전화번호: 010-XXXX-XXXX 자동 하이픈 삽입
- 초성 검색: `es-hangul` 라이브러리 (예: "ㅎㄱ" → "한길")
- UX 텍스트: 해요체 존댓말 ("출석이 완료됐어요")
- 날짜: YYYY.MM.DD, 상대시간 ("방금 전", "어제")
- 소셜 로그인 순서: 카카오 → 구글 → PIN
- 상세: `docs/KOREAN-UX.md` 참조

### 5.6 사용자 편의 UX
- 역할별 하단 네비게이션 (관리자 5탭, 참가자 4탭)
- 역할별 첫 실행 경험 (관리자: 위자드, 교사: 코치마크, 참가자: 즉시 진입)
- 모바일 바텀 시트 (상세 드릴다운)
- Pull-to-refresh (대시보드, 출석)
- 햅틱 피드백 (출석, 프로그램, 탭 전환)
- 토스트 알림: Sonner (성공 3초, 에러 수동 닫기)
- 상세: `docs/UX-PATTERNS.md`, `docs/REALTIME-UX-PATTERNS.md` 참조

### 5.7 브라우저 호환
- Chrome, Safari, **Samsung Internet** (최신 2개 버전)
- iOS Safari PWA: 홈 화면 추가 시 push 지원 (iOS 16.4+)
- Samsung Internet: 한국 시니어 사용자 주요 브라우저 (테스트 필수)
- iOS Safari, Android Chrome

## 7. 제약사항

- **비용**: 앱 제공자 비용 0원. 각 교회가 자체 Supabase 무료 티어 사용
- **백엔드**: 자체 서버 없음, BYOS(Bring Your Own Supabase) 모델
- **개발 기간**: MVP 기준 4~6주
- **유지보수**: 교회 IT 봉사자가 관리 가능한 수준의 복잡도
- **데이터 소유권**: 각 교회가 자체 Supabase에 데이터 보유 (앱 제공자 접근 불가)

## 8. BYOS (Bring Your Own Supabase) 모델

### 8.1 개요
사용자(교회 관리자)가 자신의 Supabase 프로젝트를 생성하고 앱에 연결하는 방식입니다.
상세 기술 설계는 `docs/BYOS.md`를 참조하세요.

### 8.2 온보딩 플로우
1. 앱 최초 접속 → Supabase 미연결 감지 → 온보딩 위자드 자동 표시
2. 단계별 스크린샷 가이드로 Supabase 가입 → 프로젝트 생성 → URL/Key 복사
3. 앱에 붙여넣기 → 연결 확인 → DB 스키마 자동 생성
4. 관리자 계정 설정 → 행사 만들기 시작

### 8.3 보안
- 각 교회 데이터는 해당 교회의 Supabase에만 저장
- 앱 제공자는 다른 교회의 데이터에 접근 불가
- 상세 보안 정책은 `docs/SECURITY.md` 참조
